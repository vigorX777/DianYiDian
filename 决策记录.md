# 决策记录

记录会影响后续开发方向的产品和技术决策。每条记录都应包含背景、决定和影响，避免后续重复讨论。

## D-001 - MVP 只做次数型记录

### 背景

产品最初讨论了次数型、完成型和混合型打卡。为了降低复杂度并验证菜单栏一键记录体验，第一版需要聚焦最小闭环。

### 决定

MVP 只支持一个次数型事项，不做完成型、时长型和多事项同时记录。

### 影响

- 默认事项为“喝水”。
- 左键点击菜单栏项直接加 1。
- 菜单中只保留打卡、撤销、重置、设置和退出。
- 后续多事项、日历、统计图都放入后续版本。

## D-002 - 使用原生 Swift + AppKit + SwiftUI

### 背景

这是一个 macOS 菜单栏小工具，核心诉求是低资源占用、菜单栏交互稳定、后续可打包成标准 Mac App。

### 决定

采用 Swift 原生实现。AppKit 负责菜单栏项和菜单交互，SwiftUI 负责设置窗口。

### 影响

- 菜单栏交互使用 `NSStatusItem`。
- 设置界面使用 SwiftUI 表单。
- 不引入 Electron、Tauri 或 Web 技术栈。

## D-003 - 当前阶段使用 SwiftPM 先落地

### 背景

当前机器只有 Command Line Tools，没有完整 Xcode，无法直接完成标准 Xcode Archive。

### 决定

先使用 SwiftPM 实现可编译、可运行、可验证的工程，并提供本地 `.app` 打包脚本。

### 影响

- 使用 `swift build` 做基础构建。
- 使用 `DianYiDianCoreChecks` 做核心业务校验。
- 使用 `Scripts/package-app.sh` 生成本地 `点一点.app`。
- 正式签名、Archive 和分发流程后续在完整 Xcode 环境下补齐。

## D-004 - 菜单栏项显示“进度图标 + 次数/目标”

### 背景

初版纯图标虽然进程启动成功，但用户在菜单栏中不容易确认应用是否存在。

### 决定

MVP 阶段菜单栏项采用“进度图标 + 次数/目标”的组合显示，例如 `1/8`。

### 影响

- 菜单栏项可见性优先于极简视觉。
- 后续可以在设置中增加“只显示图标 / 图标加数字”的显示模式。

## D-005 - 每次开发必须同步治理文件

### 背景

项目需要可回溯地记录每次调整，方便后续恢复上下文和定位问题。

### 决定

后续每次开发都遵循：

1. 开发前查看 `任务看板.md`。
2. 开发中保持改动聚焦。
3. 验证后更新 `开发日志.md`。
4. 如果产生产品或技术方向变化，更新 `决策记录.md`。
5. 如果影响版本行为，更新 `版本记录.md`。

### 影响

- 任何非临时改动都应有日志记录。
- 交付说明必须包含验证结果。
- 后续代理接手时优先读取这些治理文件。
