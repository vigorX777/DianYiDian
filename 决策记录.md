# 决策记录

记录会影响后续开发方向的产品和技术决策。每条记录都应包含背景、决定和影响，避免后续重复讨论。

## D-001 - MVP 只做次数型记录

### 背景

产品最初讨论了次数型、完成型和混合型打卡。为了降低复杂度并验证菜单栏一键记录体验，第一版需要聚焦最小闭环。

### 决定

MVP 只支持一个次数型事项，不做完成型、时长型和多事项同时记录。

### 影响

- 默认事项为“喝水”。
- 左键点击菜单栏项直接加 1。
- 菜单中只保留打卡、撤销、重置、设置和退出。
- 后续多事项、日历、统计图都放入后续版本。

## D-002 - 使用原生 Swift + AppKit + SwiftUI

### 背景

这是一个 macOS 菜单栏小工具，核心诉求是低资源占用、菜单栏交互稳定、后续可打包成标准 Mac App。

### 决定

采用 Swift 原生实现。AppKit 负责菜单栏项和菜单交互，SwiftUI 负责设置窗口。

### 影响

- 菜单栏交互使用 `NSStatusItem`。
- 设置界面使用 SwiftUI 表单。
- 不引入 Electron、Tauri 或 Web 技术栈。

## D-003 - 当前阶段使用 SwiftPM 先落地

### 背景

当前机器只有 Command Line Tools，没有完整 Xcode，无法直接完成标准 Xcode Archive。

### 决定

先使用 SwiftPM 实现可编译、可运行、可验证的工程，并提供本地 `.app` 打包脚本。

### 影响

- 使用 `swift build` 做基础构建。
- 使用 `DianYiDianCoreChecks` 做核心业务校验。
- 使用 `Scripts/package-app.sh` 生成本地 `点一点.app`。
- 正式签名、Archive 和分发流程后续在完整 Xcode 环境下补齐。

## D-004 - 菜单栏项显示“进度图标 + 次数/目标”

### 背景

初版纯图标虽然进程启动成功，但用户在菜单栏中不容易确认应用是否存在。

### 决定

MVP 阶段菜单栏项采用“进度图标 + 次数/目标”的组合显示，例如 `1/8`。

### 影响

- 菜单栏项可见性优先于极简视觉。
- 后续可以在设置中增加“只显示图标 / 图标加数字”的显示模式。

## D-005 - 每次开发必须同步治理文件

### 背景

项目需要可回溯地记录每次调整，方便后续恢复上下文和定位问题。

### 决定

后续每次开发都遵循：

1. 开发前查看 `任务看板.md`。
2. 开发中保持改动聚焦。
3. 验证后更新 `开发日志.md`。
4. 如果产生产品或技术方向变化，更新 `决策记录.md`。
5. 如果影响版本行为，更新 `版本记录.md`。

### 影响

- 任何非临时改动都应有日志记录。
- 交付说明必须包含验证结果。
- 后续代理接手时优先读取这些治理文件。

## D-006 - P0 日期表只做菜单内状态展示

### 背景

P0 需要在右键菜单中展示当月打卡状态，同时避免把菜单交互做得过重。

### 决定

当月日期表以内嵌菜单视图展示，只显示当前月，不支持点击日期、不支持月份切换、不打开独立历史页面。

### 影响

- 日期表使用周一到周日的 7 列网格。
- 日期点使用水位表达完成比例。
- 历史数据来自本地历史记录和今日状态。
- 后续如果要查看详情或切换月份，应升级为独立历史窗口或弹出详情，不在 P0 菜单内继续堆交互。

## D-007 - P0 音效使用系统音效

### 背景

打卡音效需要快速落地，同时避免引入音频资源版权、体积和打包问题。

### 决定

单次打卡使用系统 `Ping` 音效，首次达标使用系统 `Glass` 音效。

### 影响

- 不新增音频资源文件。
- 继续使用现有设置开关控制音效是否播放。
- 音效服务需要预加载并持有 `NSSound` 实例，避免点击时临时对象释放导致播放不稳定。
- 后续如需品牌化音效，再单独引入资源文件和版权确认。

## D-008 - 菜单栏显示模式默认保留“图标 + 数字”

### 背景

用户需要可选择更极简的菜单栏形态，但当前阶段仍要保证启动后能快速确认 App 是否存在，以及今日次数是否更新。

### 决定

设置中提供“只显示图标”和“图标 + 数字”两种模式，默认继续使用“图标 + 数字”。

### 影响

- 新用户和旧配置缺省时都会看到 `次数/目标`。
- 追求极简菜单栏的用户可以切换为只显示图标。
- 新增设置字段必须兼容旧版 UserDefaults 数据，避免升级后丢失原设置。

## D-009 - 多场景第一版只支持次数型

### 背景

场景扩展需要先解决多个打卡目标的新增、切换、菜单栏展示和历史隔离，完成型、时长型会显著增加状态和统计复杂度。

### 决定

0.2.0 只支持次数型场景。数据模型保留 `type = count`，但设置页不开放其他类型。

### 影响

- 新增场景仍使用每日目标次数和今日初始次数。
- 旧单事项数据迁移为一个默认次数型场景。
- 后续扩展完成型或时长型时，需要单独补业务状态、菜单文案和历史统计逻辑。

## D-010 - 多图标菜单栏使用固定场景

### 背景

多个场景全部显示在菜单栏会快速占用空间，但只显示一个当前场景又会降低高频场景的直接打卡效率。

### 决定

默认使用“当前场景”模式；用户可以切换到“固定场景多图标”模式，并通过每个场景的“固定到菜单栏”开关决定哪些场景显示为独立菜单栏图标。

### 影响

- 左键点击当前模式图标时，只给当前场景打卡。
- 左键点击固定场景图标时，只给对应场景打卡。
- 未固定或停用的场景不占用菜单栏，但仍可在右键菜单和设置页管理。

## D-011 - 场景图标第一版使用 SF Symbols

### 背景

场景需要更丰富的图标表达，但上传图片会引入裁剪、透明度、深浅色适配和资源管理问题。

### 决定

第一版只提供内置 SF Symbols 图标，不支持上传自定义图标。

### 影响

- 图标在菜单栏和设置页都使用系统符号名渲染。
- 图标主题色跟随场景配置。
- 自定义图片和 Emoji 图标留到后续版本评估。
